## TASK-1035: move process.exit() to main entry point: commands return exit codes
**Meta:** P2 | M | READY | Focus:no | 2-code-generation | local | cli/src/main/ts/index.ts, cli/src/main/ts/application/commands/

**Depends:** TASK-1033

### Context

At least 14 command files call `process.exit()` directly. This makes commands untestable (process.exit() terminates the test runner), prevents the dispatcher from handling errors uniformly, and makes it impossible to compose commands. The fix is standard: commands return an exit code (number) or throw a typed error; `index.ts` calls `process.exit()` once, on the result. `TASK-1033` (command routing into registry) is a prerequisite because it clarifies the dispatcher's contract before this task changes what commands return.

### Acceptance Criteria

- [ ] Command interface (or base type) specifies that `execute()` returns `Promise<number>` where the number is the exit code (0 = success).
  - `file: cli/src/main/ts/application/commands/` (whichever file defines the Command interface)

- [ ] All commands that previously called `process.exit(N)` now return `N` from `execute()` instead.
  - `cmd: grep -rn "process\.exit" cli/src/main/ts/application/commands/ | wc -l` returns 0

- [ ] `index.ts` calls `process.exit()` exactly once, using the return value from the dispatcher.
  - `file: cli/src/main/ts/index.ts`
  - `cmd: grep -c "process\.exit" cli/src/main/ts/index.ts` returns 1

- [ ] `command-dispatcher.ts` propagates the exit code returned by commands without swallowing it.
