### Context

**ADR-026 resolution:** escalations.jsonl currently functions as a Layer 1 event log under ADR-026. Upsert semantics make it a mutable Layer 2 state projection — an intentional deviation. The rationale: escalation records are operational state, not audit events. The append-only model was chosen by default, not by design decision, and produces unbounded phantom chains (350+ records, ~15 actionable signals). This task makes the deviation explicit by treating escalations.jsonl as a state store, not an event log. No new ADR is required — this is a corrective reclassification within the existing layer model.

**Implementation guidance:** `EscalationStore` at `cli/src/main/ts/application/use-cases/escalation-store.ts` already owns the write path. A `compact()` method exists and already deduplicates OPEN records by `${type}::${subject}` key. The upsert change means:
- `append()`: read the full file, find any existing OPEN record with the same key; if found, update its `reason` and `timestamp` in place and rewrite; if not found, append as before.
- `resolve()`: find the OPEN record by (type, subject) or escalation_id, set `status: "RESOLVED"` and `resolved_at` in place, rewrite — no tombstone append.
- Subject normalization: apply `subject.trim().toLowerCase()` before key construction in both read and write paths.

The existing `compact()` method can be simplified or removed once upsert is in place — it was a workaround for the append-only model. The `getOpen()` method's resolved-set logic (filtering by matching escalation_id) also simplifies to a direct `status === 'OPEN'` filter.
