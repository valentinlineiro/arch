## TASK-1059: govern: always emit REVIEW_REQUEST entries for REVIEW-status tasks after each tick

### Context

Promoted from IDEA-inbox-stable-ledger-invariant.

`arch govern`'s INBOX regeneration pass overwrites the full INBOX, discarding manually-written REVIEW_REQUEST entries for tasks still in REVIEW status. The task file retains its REVIEW status, but the INBOX signal for the human Auditor is silently lost. The next govern tick produces an INBOX with no pending review entries, even though tasks are waiting.

The fix: after the hygiene pass, govern scans `docs/tasks/` for all REVIEW-status tasks and ensures each has a corresponding `## [AWAITING_REVIEW] TASK-XXX` section in INBOX. This is idempotent — if the entry already exists (correct format), it is not duplicated. Missing entries are appended.

This is complementary to TASK-1058 (hygiene removes stale entries for archived tasks). Together they make INBOX complete and accurate after every tick.

### Acceptance Criteria

- [x] After a govern tick, every task in REVIEW status has a corresponding `## [AWAITING_REVIEW] TASK-XXX` entry in INBOX.md. → file: cli/src/main/ts/application/use-cases/govern-system.ts
- [ ] If the entry already exists, a second govern tick does not duplicate it. → prose: unit test: two consecutive ticks produce exactly one AWAITING_REVIEW entry per REVIEW-status task
- [ ] A task that moves from REVIEW to DONE (archived) has its AWAITING_REVIEW entry removed on the next tick (existing hygiene via TASK-1058 covers this). → prose: confirmed by inbox-hygiene.test.ts existing tests
