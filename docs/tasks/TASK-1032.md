## TASK-1032: fix enforcement/analysis boundary: move SubprocessRunner call from govern-system.ts to govern-command.ts
**Meta:** P1 | S | READY | Focus:no | 2-code-generation | local | cli/src/main/ts/application/use-cases/govern-system.ts, cli/src/main/ts/application/commands/govern-command.ts

**Depends:** none

### Context

ARCH's core constitutional invariant: deterministic enforcement never mixes with LLM analysis. `govern-system.ts` (a use-case) currently calls `SubprocessRunner.runSync('./scripts/arch.sh', ['analyze'])` — a side-effect from the use-case layer into the CLI layer that triggers LLM analysis. This is the only place in the codebase where this boundary is violated. The fix is not architectural redesign: move the SubprocessRunner call to govern-command.ts (the command layer, which is the appropriate place for CLI-layer side effects) and have govern-system.ts signal intent via its return value instead of invoking it directly.

- [ ] `govern-system.ts` contains no reference to `SubprocessRunner` and no call to `arch analyze` or `arch.sh`.

- [ ] `govern-command.ts` handles the `arch analyze` invocation as a post-govern step, triggered by the return value or a flag from `GovernSystem.execute()`.
  - `file: cli/src/main/ts/application/commands/govern-command.ts`

- [ ] The analyze step is skipped when `arch govern` is called with `--no-analyze` or equivalent flag, making it opt-out rather than always-on.
