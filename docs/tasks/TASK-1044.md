## TASK-1044: property-based tests for meta-line parser: isomorphism and boundary coverage
**Meta:** P2 | M | READY | Focus:yes | 2-code-generation | local | cli/src/test/ts/

**Depends:** TASK-1039

### Context

The meta line (`**Meta:** P0 | S | READY | Focus:no | 2-code-generation | local | docs/tasks/`) is the source of truth for task state in ARCH. `MarkdownTaskRepository.parseTask` parses it; serialization writes it back. If parsing is wrong, governance is wrong — the wrong task status propagates through every govern tick. Current tests verify known-good inputs. They don't verify the parser is a correct inverse of the serializer, and they don't verify behavior on boundary or malformed inputs. A round-trip property test (`parse(serialize(task)) === task`) catches parser regressions that unit tests miss because it doesn't depend on the author anticipating the specific failure mode.

### Acceptance Criteria

- [ ] A round-trip property test verifies `parse(serialize(task)) === task` for all valid task field combinations (all priority values × all sizes × all statuses × Focus:yes/no).
  - `file: cli/src/test/ts/meta-line-parser.test.ts` (new file)
  - `prose: test iterates over the full valid value space for priority, size, status, focus and asserts round-trip equality`

- [ ] Boundary tests cover: empty context field, context field at maximum length, special characters in task title (colons, dashes, pipes), Unicode in title.
  - `prose: at least 4 boundary test cases in the file`

- [ ] Malformed input tests cover: missing fields, wrong field count, unrecognized status value, wrong pipe separator. Parser must not throw — it should return null or a typed error.
