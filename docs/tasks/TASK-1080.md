### Context

ARCH has ~15 active subsystems. A new user hits all of them simultaneously on first use. The smartcart-os pilot confirmed: the first friction is Hansei validation on a task the user considers done, before they've seen any governance value.

Two mechanisms are needed:

**1. archProfile** — a named preset that configures all subsystems at once. Three built-in profiles:

- `minimal` — task lifecycle only. Hansei advisory (not blocking). No corpus, no sprint, no drift structural/governance groups. Fastest onboarding.
- `standard` — task lifecycle + Hansei blocking + DriftChecker TaskHealth group + sprint management. Reasonable default for teams.
- `full` — everything enabled and enforced (current default, backward compatible).

**2. modules block** — fine-grained per-subsystem control. Overrides profile defaults:

```json
{
  "archProfile": "minimal",
  "modules": {
    "hansei": "advisory",         // advisory | blocking | disabled
    "sprint": "disabled",         // enabled | disabled
    "corpus": "disabled",         // enabled | disabled
    "driftChecks": ["TaskHealth"],// which DriftChecker groups to run
    "coreFlows": "disabled"       // enabled | disabled
  }
}
```

**The enforcement vs instrumentation distinction:**
- `blocking` — violations prevent close/review from passing
- `advisory` — violations are surfaced as warnings but do not block
- `disabled` — subsystem does not run at all

Hansei at `advisory` still runs and produces output — the retrospective value is preserved without the close-path friction. This is the right entry point for new adopters.
