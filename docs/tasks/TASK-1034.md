## TASK-1034: standardize CLI output strategy: one path, not three
**Meta:** P2 | S | READY | Focus:no | 2-code-generation | local | cli/src/main/ts/application/commands/

**Depends:** none

### Context

Commands currently use three different output mechanisms: `console.log`, `process.stderr.write`, and the `output-formatter.ts` utility. No command is wrong individually — the problem is that the three paths are inconsistent, making it impossible to redirect or suppress output reliably and harder to test. The fix is to pick one strategy and route all command output through it. The output-formatter utility already exists as the right abstraction — it just isn't consistently used.

### Acceptance Criteria

- [ ] All command files route user-facing output through `output-formatter.ts` (or the chosen single output abstraction). No direct `console.log` in command files.
  - `cmd: grep -rn "console\.log" cli/src/main/ts/application/commands/ | wc -l` returns 0

- [ ] Error output (failures, validation errors) routes through `process.stderr` or an equivalent error channel in the formatter — not `process.stdout`.
  - `prose: running a failing command and piping stdout to /dev/null shows no error text on stdout`

- [ ] The output formatter (or chosen abstraction) is injectable in tests — commands do not construct their own formatter instance inline.
  - `prose: at least one command test asserts on captured output via the formatter mock/stub`

- [ ] `npm test` passes.
  - `prose: all existing tests pass`

- [ ] `arch review` passes.
  - `cmd: node cli/dist/index.js review`
