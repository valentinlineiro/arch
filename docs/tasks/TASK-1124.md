## TASK-1124: ANDON_HALT stops the autonomous loop: exit non-zero when detected
**Meta:** P1 | S | READY | Focus:no | 1-code-reasoning | local | cli/src/main/ts/application/commands/govern-command.ts, cli/src/main/ts/application/use-cases/govern-system.ts

**Depends:** TASK-1123

### Context

ANDON_HALT currently writes to NOTIFICATIONS.md and continues. The loop reads the next task and starts it. The escalation mechanism is toothless in autonomous mode.

Fix: arch govern exits non-zero when ANDON_HALT entries exist in NOTIFICATIONS.md. The loop orchestrator (GitHub Actions, cron, claude task loop) treats non-zero govern as a halt requiring human intervention.

This is the circuit breaker. Everything else (stale tasks, max attempts, broken environment) escalates to ANDON_HALT. ANDON_HALT stops the loop. A human clears it by deleting the entry from NOTIFICATIONS.md and running arch govern manually.

  - `cmd: echo "[ANDON_HALT] test" >> docs/NOTIFICATIONS.md && node cli/dist/index.js govern; echo "exit: $?"`
- [ ] arch govern prints a clear halt message naming the ANDON_HALT entry before exiting
  - `prose: ANDON_HALT: TASK-XXX exceeded max attempts — remove from NOTIFICATIONS.md to resume`
- [ ] arch govern exit 0 when no ANDON_HALT entries exist (normal operation unchanged)
  - `prose: arch govern on a clean repo exits 0 as before`
- [ ] arch govern --force bypasses the ANDON_HALT check for human-initiated runs
  - `prose: arch govern --force runs normally even with ANDON_HALT entries`
