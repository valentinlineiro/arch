## TASK-1075: Primary user flow gate when AC stubs an endpoint
**Meta:** P1 | S | READY | Focus:yes | 1-code-reasoning | claude-code | cli/src/main/ts/application/use-cases/drift-checker.ts, docs/PROJECT.md

**Depends:** none

### Context

Source: IDEA-stub-e2e-coverage-prompt.md

The smartcart-os incident: TASK-008 stubbed POST /api/inventory (correct per cut list). POST /api/inventory was the main CTA. The app was broken. ARCH's close path verified the stub was in place but never checked whether the stubbed path was on a primary user flow. A user discovered the breakage.

The fix: when a task stubs an endpoint or removes a capability, arch review checks whether that path appears in `docs/PROJECT.md ## Core Flows`. If yes, it emits a FLOW-REGRESSION warning.

### Acceptance Criteria

- [ ] `arch review` emits `[FLOW-REGRESSION] <path> is referenced in Core Flows but was stubbed or removed` when a stubbed path matches a Core Flow predicate.
  - `prose: stubbing POST /api/inventory with Core Flows containing /api/inventory triggers FLOW-REGRESSION`

- [ ] Warning does not block the close path: it is advisory only.
  - `prose: task with FLOW-REGRESSION warning can still close`

- [ ] If PROJECT.md has no Core Flows section, the check is a no-op.
  - `prose: repos without Core Flows section emit no FLOW-REGRESSION warnings`

- [ ] `npm test` passes.
  - `prose: 707+ tests pass`
