The smartcart-os incident: TASK-008 stubbed POST /api/inventory (correct per cut list). POST /api/inventory was the main CTA. The app was broken. ARCH's close path verified the stub was in place but never checked whether the stubbed path was on a primary user flow. A user discovered the breakage.

The fix: when a task stubs an endpoint or removes a capability, arch review checks whether that path appears in `docs/PROJECT.md ## Core Flows`. If yes, it emits a FLOW-REGRESSION warning.

## Hansei
**Severity:** H0
**Category:** [SpecDrift]
**Decision:** checkFlowRegressionRisk() added to StructuralChecker. Parses Core Flows section of PROJECT.md for endpoint predicates (POST/GET/PUT/DELETE paths). Cross-references open tasks for stub-404 patterns on those paths. Emits WARN with [FLOW-REGRESSION] entries if a stub task references a core flow path. No-op when PROJECT.md absent or no endpoint predicates. 707 tests pass.
**Constraint:** Detection is text-based (grep for 404 patterns in task content + path match). Does not inspect runtime code. False negatives possible if stub is implicit. Acceptable for advisory warning.
**Cost:** Extra readDirectory + readFile per open task on each arch review. Negligible on small repos; may be slow with 100+ open tasks.
**Forward Action:** Consider caching core flow paths between check runs if review becomes slow.
