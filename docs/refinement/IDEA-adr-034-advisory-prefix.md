# IDEA: ADR-034 advisory prefix compliance — THINK INBOX entries must use [ADVISORY]

**Status:** DRAFT
**Created:** 2026-06-02
**Source:** Phase-2.5 — ADR-034 Rule 3 compliance audit
**Candidate-class:** 6-writing
**Candidate-size:** XS

## Problem

ADR-034 Rule 3 mandates that non-deterministic INBOX entries generated by `arch analyze` (including THINK) must be prefixed `[ADVISORY]`. The INBOX entry classification table (ADR-034) explicitly lists `[ADVISORY]` as the prefix for `analyze-command.ts` output. However, THINK's current INBOX regeneration produces `[REPLENISHMENT]`, `[RECLAIMED]`, and other blocks without the required `[ADVISORY]` prefix.

This is an ADR compliance gap. Without the prefix, readers and automated tools cannot distinguish deterministic governance signals from advisory THINK output.

## Proposed fix

Update `docs/agents/THINK.md` Phase 1 INBOX regeneration step (step 4) to require:

> All THINK-generated INBOX blocks must carry `[ADVISORY]` prefix. Specifically: `[REPLENISHMENT]` → `[ADVISORY] [REPLENISHMENT]`, `[RECLAIMED]` → `[ADVISORY] [RECLAIMED]`. Deterministic blocks (those written by `arch govern` — e.g., `[ANDON_HALT]`, `[PATTERN-ALERT]` from hansei-synthesizer) are exempt and retain their existing prefix.

This preserves the informational content of each block while making the source layer explicit.
