# IDEA: governance-knowledge-store — structured knowledge that governs AI and human decisions
**Decision-required:** yes
**Created:** 2026-05-22
**Source:** Human-led design session
**Status:** DEFERRED
**Meta:** P1 | XL | cli | context

## Problem

ARCH accumulates operational memory (tasks, signals, causal edges, tensions) but has no structured store for **governance-eligible knowledge** — claims that have passed human validation and can authoritatively influence both AI and human decisions.

Current state: knowledge is implicit across 5+ artifact types, no single query surface, and no authority tiering. Both `arch govern` and `arch analyze` lack a deterministic source for "what does this system know about its own domain that it can trust?" This violates the intent of IDENTITY.md §7 — LLMs must never govern, but currently there is no alternative source for governance knowledge that is not itself LLM-derived.

## Proposed solution

A **directory-backed knowledge store** at `.arch/knowledge/` with a rebuildable index. Every claim is a separate file. The index is the query surface for both deterministic enforcement and LLM analysis.

### Structure

```
.arch/knowledge/
├── live/
│   ├── K-001-diffs-core-adapters.json
│   └── K-002-refactor-classification.json
├── proposed/
│   └── ...
├── decayed/
│   └── ...
└── .index              ← rebuildable bitmap (ephemeral, never committed)
```

### Claim schema

```json
{
  "id": "K-001",
  "created": "2026-05-22",
  "type": "invariant",
  "claim": "diffs_touching_core_and_adapters_must_be_split",
  "rationale": "PR reviews for combined diffs take 3x longer and miss edge cases",
  "evidence": ["TASK-101", "TASK-204", "TASK-312"],
  "authority": "HUMAN",
  "status": "active",
  "supersedes": null,
  "decay": "2026-08-22"
}
```

### Authority tiers

| Tier | Stored in | Readable by | Governable by |
|------|-----------|-------------|---------------|
| `HUMAN` | `live/` | `arch govern`, `arch analyze`, human | Yes — enforceable |
| `PROPOSED` | `proposed/` | `arch analyze`, human | No — suggestion only |
| `DECAYED` | `decayed/` | human (audit) | No |

### Query surface: `.index`

Format: flat JSON mapping claim IDs to shard path and filterable fields:

```json
{
  "K-001": {
    "path": "live/K-001-diffs-core-adapters.json",
    "type": "invariant",
    "status": "active",
    "authority": "HUMAN"
  }
}
```

Regenerated by `arch govern --rebuild-index` or automatically on startup (`<100ms` for any plausible corpus). Ephemeral — never committed, never conflicts.

### Consumption pipeline

```
Signal → Chronicle Arbitration → Knowledge Proposal (PROPOSED) → Human Promotion → Knowledge Claim (HUMAN, live/) → Governance Enforcement
```

`arch govern` reads only `live/` with `authority: HUMAN`. `arch analyze` reads everything including `proposed/` for suggestions. `arch knowledge query` provides a human CLI over the index.

### Duplicate detection

Content-addressable filenames: `K-{hash-of-claim}.json`. Same claim from different sources hashes to the same filename — second writer detects collision and appends evidence to the existing claim instead of creating a duplicate.

### Compaction = file moves

- **Decay**: move from `live/` to `decayed/`
- **Supersession**: add `supersedes: "K-001"` field, move old to `decayed/`, new stays in `live/`
- **No rewriting**: audit trail is the directory tree itself

### Decay protocol

Every claim gets a `decay` timestamp (default 90 days). `arch govern` checks daily: expired claims are moved to `decayed/` and flagged for human re-validation. If no re-validation occurs, the claim is permanently removed on the next compaction cycle.

## Dependencies

- Phase P (Productization) has established `.arch/` as the operational data directory
- Causal signal arbitration (Chronicle) is the upstream — knowledge proposals originate from validated signal patterns

XL — touches CLI domain models, repository layer, governance enforcement, analysis pipeline, and human query surface.
