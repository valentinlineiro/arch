# ADR-023: Deterministic Gate Invariant

## Status
PROPOSED

## Context
ARCH is a governing system. To maintain epistemic integrity, the boundary between machine authority (governance) and heuristic analysis (advisory) must be structurally absolute. Relying on probabilistic outputs (LLMs) for governance gates introduces non-reproducibility and allows the system to drift into "informal enforcement" where heuristic suggestions are treated as operational facts.

## Decision
We establish the **Deterministic Gate Invariant**: All governance-bearing outputs must be generated by deterministic code. Probabilistic or heuristic outputs are strictly advisory and possess zero machine authority.

### 1. Classification of Channels
Every ARCH output must belong to exactly one of three channels, with strictly defined authority:

| Channel | Source | Authority | Examples |
| :--- | :--- | :--- | :--- |
| **Deterministic** | Code/Regex/Invariants | **Authoritative.** Controls exit codes, state transitions, and CI outcomes. | `arch review`, `arch govern` (core), `arch task review` predicates. |
| **Advisory** | LLM/Heuristics/Inference | **Non-Authoritative.** Human-facing only. May influence attention, never state. | `arch reflect` (Tier 2), `arch ask` (relevance ranking), `arch capture` (drafts). |
| **Reporting** | Materialized Projection | **Derivative.** Strictly non-authoritative. One-way mirror. | `README.md`, `ROADMAP.md`, `docs/METRICS.md`. |

### 2. Operational Rules
*   **Zero Machine Authority for Advisory:** No advisory output may directly or indirectly produce a non-zero exit code, a `BLOCKED` task status, or an `ANDON_HALT` trigger.
*   **The Consumption Prohibition:** No component of the ARCH system may consume an Advisory or Reporting output as operational evidence for a subsequent Deterministic gate.
*   **Reproducibility Requirement:** For any given observable repository state, a Deterministic gate must yield an identical outcome every time it is executed.
*   **Degradation Policy:** A failure in the Advisory channel (e.g., LLM timeout, API error) is an informational event only and must never result in a governance failure or block the execution pipeline.
*   **Exit Code Hardening:** `reflect-command.ts` and similar entry points must ensure that Tier 2 (Advisory) analysis always returns exit code 0. Only Tier 1 (Deterministic) validators have exit code control.

### 3. Implementation Contract: `reflect-command.ts`
The `reflect` subcommand must be restructured to enforce these rules:
1.  **Stage 1 (Deterministic):** Runs diff-based Hansei validators. These control the exit code.
2.  **Stage 2 (Advisory):** Runs LLM analysis. This block must be wrapped in a `try/catch` that suppresses errors and **always** proceeds to `process.exit(0)` if Stage 1 passed.
3.  **Serialization:** Advisory findings must be serialized separately from the deterministic validator state to prevent accidental merging.

## Consequences
*   **Epistemic Stability:** Prevents the system from "inventing" continuity or generating apparent consistency over ambiguous data.
*   **Predictable CI:** Repositories governed by ARCH will have stable, reproducible build outcomes.
*   **Auditable Traceability:** Every governance decision (PASS/FAIL) will have a deterministic causal trace to a specific invariant or code block.
*   **Human Sovereignty:** Explicitly reserves "Soft Pressure" (Advisory) for human attention, ensuring the machine never oversteps its deterministic authority.
